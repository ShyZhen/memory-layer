# Memory Layer Plugin

> https://github.com/ShyZhen/memory-layer

**An enterprise-grade multi-channel layered memory plugin for multi-user OpenClaw agents.**

`memory-layer` adds layered memory for multi-user OpenClaw agents. It separates shared team memory from per-user personal memory, prevents cross-user memory leakage, and preserves compatibility with legacy `MEMORY.md` / `memory/*.md` habits.

By default, it is channel-agnostic and works from OpenClaw session keys, making it suitable for DingTalk, Telegram, and other channels with isolated session keys.

For the Chinese version, see [README.md](./README.md).
For maintenance and design details, see [ARCHITECTURE.md](./ARCHITECTURE.md).

## Core Capabilities

- Separates shared memory from per-user personal memory
- Preserves compatibility with legacy `MEMORY.md` / `memory/*.md` habits
- Preserves OpenClaw's native "long-term memory vs daily notes" split, but remaps it into a per-user multi-user model
- Prevents new personal memory from being written back into shared legacy files
- Isolates DM memory per user without requiring one agent or one workspace per end user
- Supports sharing one team memory file across multiple agents
- Can be restricted to specific channels through `enabledChannels`

## Typical Use Cases

- One agent serves multiple DingTalk users at the same time
- Each user needs their own durable long-term memory
- Team knowledge should remain reusable across users and sessions
- New personal memory should no longer flow back into legacy shared `MEMORY.md`
- A gradual migration is needed from old memory habits to a layered memory model

## Storage Layout

Default file layout:

- `shared/memory.md`: team/shared memory
- `users/<channel>/<account>/<peer>/memory.md`: durable personal memory for the current user
- `users/<channel>/<account>/<peer>/notes/YYYY-MM-DD.md` or `YYYY-MM-DD-*.md`: redirected legacy daily notes for the current user
- `users/<channel>/<account>/<peer>/history/YYYY-MM-DD.md`: recent conversation history for the current user

By default, these files live under `.memory-layer/` inside the agent workspace.

## How It Works

Before each agent run, the plugin injects layered context according to `contextInjectionMode`:

- shared memory
- personal memory for the current user
- compatibility `notes/` for the current user
- recent `history/` for the current user, but only when the injection policy allows it

After each run, the plugin automatically:

- appends the latest real user message and the current assistant reply to the current user's `history` file
- saves explicit memory commands when enabled

Important:

- When `contextInjectionMode = "new-session"`, shared memory, personal memory, and compatibility `notes/` are injected on the first turn of a new session
- In that mode, `history/` is only injected for sessions reopened after automatic archival, not after user-triggered `/new` or `/reset`
- The plugin does not archive the synthetic startup prompt generated by `/new` or `/reset` into `history/`

For plugin-managed DM sessions, it also intercepts and rewrites legacy memory file reads and writes so that new personal memory no longer flows back into shared legacy `MEMORY.md`.
For plugin-managed `shared/memory.md`, personal `memory.md`, and daily `history`, it also applies soft stored-size caps so those files do not grow without bound.

You can think of this as a multi-user optimization of OpenClaw's native memory model rather than a replacement for it:

- logic that would previously write to `MEMORY.md` / `memory.md` is redirected into the **current user's personal long-term `memory.md`**
- logic that would previously write to `memory/YYYY-MM-DD.md` is redirected into the **current user's `notes/`**
- this preserves the native "long-term memory vs daily notes" split while avoiding cross-user mixing in one shared file set
- Explicit Save Commands are also supported; see the section below.

## Recommended Config

**After enabling the plugin, disable `session-memory` so OpenClaw does not keep writing into the default legacy memory files.**

```json
{
  "hooks": {
    "internal": {
      "entries": {
        "session-memory": {
          "enabled": false
        }
      }
    }
  },
  
  "plugins": {
    "allow": [
      "memory-layer"
    ],
    "entries": {
      "memory-layer": {
        "enabled": true,
        "config": {
          "baseDir": ".memory-layer",
          "recentHistoryDays": 2,
          "includeGroups": false,
          "allowInlineSaveCommands": true,
          "contextInjectionMode": "new-session"
        }
      }
    }
  }
}
```

## Config Reference

- `baseDir`
Root directory for layered memory. Relative paths are resolved from the current agent workspace. Default: `.memory-layer`.
- `sharedFilePath`
Optional path to the shared memory file.
If omitted, the default is `<baseDir>/shared/memory.md`.
If multiple agents should share one team memory file, point all of them to the same absolute path.
- `recentHistoryDays`
How many most-recent dated history/note buckets with actual files to inject, rather than simply looking back by calendar day. Default: `2`.
- `contextInjectionMode`
Unified policy for the whole `Layered Memory Context`, including shared memory, personal memory, compatibility notes, and recent history. Default: `new-session`.
Supported values:
  - `always`: inject on every turn, preserving legacy behavior
  - `new-session`: shared memory, personal memory, and compatibility notes are injected on the first turn of a new session; `history/` is only injected for sessions that were reopened after automatic archival, not after user-triggered `/new` or `/reset`. If shared or personal memory was just saved in the current session, it is injected once more on the next turn. This is the default and usually fits chat workflows better
  - `off`: disable the entire layered context injection
- `historyInjectionMode`
Deprecated compatibility alias. Existing configs can still use it, but new setups should prefer `contextInjectionMode`.
- `includeGroups`
Whether the plugin should also handle groups, channels, threads, or other non-DM scopes. Default: `false`.
- `enabledChannels`
Optional channel allowlist.
If omitted, the plugin works for all supported channels.
- `autoCreateFiles`
  Whether the plugin should pre-create the default layered memory files and template content. Enabled by default.
  When disabled, the plugin will not proactively create default `shared/memory.md`, personal `memory.md`, or `meta.json` templates, but real writes may still create required parent directories or target files.
- `allowInlineSaveCommands`
Whether explicit save commands are enabled. Enabled by default.
- `enabledAgents`
Optional allowlist of agent IDs.
If omitted, the plugin applies to all agents that load it.
- `maxStoredSharedChars`
Soft size cap for the stored shared memory file. Default: `20000`. Use `0` to disable trimming.
- `maxStoredPersonalChars`
Soft size cap for each user's stored personal `memory.md`. Default: `20000`. Use `0` to disable trimming.
- `maxStoredHistoryChars`
Soft size cap for each user's daily `history/YYYY-MM-DD.md`. Default: `30000`. Use `0` to disable trimming.

## Explicit Save Commands

If a user message contains one of the following commands, the plugin will persist it automatically:

- `记住：...`
- `remember: ...`
- `共享记忆：...`
- `remember-shared: ...`

Rules:

- `记住` / `remember` writes to the current user's `memory.md`
- `共享记忆` / `remember-shared` writes to the shared memory file

This part can be seen as an additional optimization on top of OpenClaw's native memory behavior:

- native legacy-path writes are still supported and redirected into the layered structure
- in addition, users can explicitly persist something immediately with `记住：...` / `remember: ...` or `共享记忆：...` / `remember-shared: ...`
- this makes "remember this now" workflows more reliable and controllable in enterprise multi-user setups

## What `notes/` And `history/` Are For

- `history/` stores the latest raw inbound user message and the matching assistant reply for short-term continuity
- When `contextInjectionMode = "new-session"`, shared memory, personal memory, and compatibility notes are injected on the first turn of a new session, while `history/` is only injected for sessions reopened after automatic archival, not after user-triggered `/new` or `/reset`
- If shared or personal memory was just saved in the current session, the layered context is injected once more on the next turn so fresh memory is immediately visible
- `notes/` is not written on every conversation turn
- It is only used when legacy tools read or write `memory/YYYY-MM-DD.md` or `memory/YYYY-MM-DD-*.md`, which the plugin redirects into the current user's private `notes/`
- If the agent no longer uses legacy `memory/*.md` paths, `notes/` may stay empty for long periods or contain only a few compatibility leftovers

## Supported Session Keys

The plugin currently supports these common OpenClaw session key shapes:

- `agent:<agentId>:dm:<peerId>`
- `agent:<agentId>:<channel>:dm:<peerId>`
- `agent:<agentId>:<channel>:<accountId>:dm:<peerId>`
- `agent:<agentId>:<channel>:direct:<peerId>`
- `agent:<agentId>:<channel>:group:<id>`
- `agent:<agentId>:<channel>:channel:<id>`
- `agent:<agentId>:<channel>:group:<id>:topic:<topicId>`
- `...:thread:<threadId>`

Important limitation:

- `session.dmScope = "main"` is allowed, but it degrades to **single-user mode**

That mode collapses all DMs into one shared session key, so the plugin can only treat all DMs as one shared "user" layer instead of isolating personal memory per real user.

## Shared Memory Across Multiple Agents

If multiple agents should share one team memory file, point all of them to the same absolute `sharedFilePath`.

Example:

```json
{
  "plugins": {
    "entries": {
      "memory-layer": {
        "enabled": true,
        "config": {
          "baseDir": ".memory-layer",
          "sharedFilePath": "C:\\Users\\zhenhuaixiu\\.openclaw\\memory\\team-shared.md",
          "recentHistoryDays": 2,
          "includeGroups": false
        }
      }
    }
  }
}
```

This means:

- each agent still keeps its own per-user memory under its own workspace
- all enabled agents read and write the same shared team memory file

## Restricting To Specific Channels

If you only want the plugin to run for selected channels, configure `enabledChannels`.

Example:

```json
{
  "plugins": {
    "entries": {
      "memory-layer": {
        "enabled": true,
        "config": {
          "enabledChannels": ["dingtalk", "telegram"],
          "baseDir": ".memory-layer"
        }
      }
    }
  }
}
```

If `enabledChannels` is omitted, the plugin applies to all supported channels.

## Deployment And Compatibility Notes

- It is strongly recommended to disable the bundled `session-memory` hook, so the old path does not continue writing personal memory into shared locations
- Legacy root `MEMORY.md` can remain as compatibility/shared context
- There is no need to point `sharedFilePath` at root `MEMORY.md`
- For plugin-managed DM sessions, do not rely on legacy `memory_search` and `memory_get`
- OpenClaw's native memory flush and search indexing target root `MEMORY.md` and `memory/*.md`, not `.memory-layer/**`
- On startup, the plugin emits warnings when `session.dmScope = "main"` or `session-memory` is still enabled, making fallback or unsafe setups easier to spot

## Notes

- This implementation is intentionally file-based and transparent for debugging and migration
- Recent history is injected as recency context, not as the sole source of truth
- For rewrite rules, compatibility rationale, and maintenance details, see [ARCHITECTURE.md](./ARCHITECTURE.md)
